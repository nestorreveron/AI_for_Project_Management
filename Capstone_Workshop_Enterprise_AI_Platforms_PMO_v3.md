
# Capstone Workshop Avanzado  
## Implementación gobernada de Enterprise AI Platforms para PMO

**Curso:** AI for Project Management II  
**Duración sugerida:** 2 horas  
**Formato:** Individual + demostración guiada del instructor  
**Nivel:** Avanzado / PMO Executive  
**Caso ficticio:** NovaCorp AI Work Platform Program  
**Resultado esperado:** Cada participante termina con un mini dossier ejecutivo para implementar una plataforma empresarial de IA y capacidades de IA para PMO.

---

# 1. Propósito del workshop

Este workshop resume todo lo trabajado durante el curso y lo lleva a un escenario realista:

> Una organización quiere implementar herramientas empresariales de IA —como Microsoft 365 Copilot, Glean, Gemini Enterprise, ChatGPT Enterprise, Claude Enterprise, Claude Cowork, Microsoft 365 Copilot Cowork, Microsoft Scout, Kiro y servidores MCP— para mejorar la forma en que su PMO trabaja, decide, reporta, aprende y gobierna.

La idea no es hacer una comparación comercial de herramientas.

La idea es que los participantes diseñen el **proyecto de implementación, adopción y gobierno** de una plataforma empresarial de IA para una PMO.

El entregable no será solo “elegir una herramienta”.

El entregable será un **dossier ejecutivo** que combine:

- Strategy Alignment.
- AI Business Case.
- Selección de herramientas.
- Arquitectura funcional.
- Casos de uso PMO.
- CPMAI Lifecycle.
- PMBOK Governance.
- DMN Decision Model.
- AI Risk Register.
- Responsible AI Controls.
- Governance Gates.
- Adoption Plan.
- Value Realization Plan.
- Recomendación ejecutiva para Steering Committee.

# 2. Mensaje central

> La PMO del futuro no solo usa IA.  
> Implementa plataformas empresariales de IA, diseña capacidades inteligentes, gobierna agentes, protege datos, modela decisiones y mide valor.

---

# 3. Por qué este taller es diferente

Muchas organizaciones están comprando o probando herramientas de IA empresarial, pero fallan en cuatro puntos:

1. Se enfocan en la herramienta antes que en el problema.
2. Conectan datos sin suficiente gobierno.
3. Lanzan agentes sin límites claros de acción.
4. No miden si el valor prometido realmente se capturó.

Este taller obliga a pensar como una PMO ejecutiva:

- ¿Qué problema queremos resolver?
- ¿Qué plataforma o combinación de plataformas tiene sentido?
- ¿Qué casos de uso PMO priorizamos?
- ¿Qué datos vamos a conectar?
- ¿Qué permisos, controles y límites necesitamos?
- ¿Qué decisiones va a apoyar la IA?
- ¿Qué acciones requieren aprobación humana?
- ¿Cómo medimos adopción, riesgo, beneficio y ROI?
- ¿Qué evidencia quedará para auditoría?

# 17. Paso 2 — Strategy Alignment

## Por qué se hace

Una implementación de Enterprise AI Platform debe tener una razón estratégica.

No se justifica por “tener IA”. Se justifica por mejorar productividad, conocimiento, decisiones, trazabilidad, riesgo y valor.

## Qué debe hacer el participante

Responder:

1. ¿Qué objetivo estratégico soporta?
2. ¿Qué dolor actual reduce?
3. ¿Qué decisiones mejora?
4. ¿Qué valor espera capturar?
5. ¿Qué nivel PPPM impacta?

## Resultado esperado

```text
Objetivo estratégico:
Dolores que reduce:
Decisiones que mejora:
Nivel PPPM impactado:
Valor esperado:
```

## Ejemplo

```text
Objetivo estratégico:
Convertir la PMO en una función más predictiva, trazable y basada en conocimiento.

Dolores que reduce:
Reporting manual, conocimiento disperso, riesgos tardíos y baja medición de beneficios.

Decisiones que mejora:
Qué riesgo escalar, qué reporte enviar, qué iniciativa priorizar y qué beneficio revisar.

Nivel PPPM:
Portfolio, Program y Project.

Valor esperado:
Menos tiempo manual, mejor toma de decisiones y mayor evidencia de valor.
```

# 22. Paso 7 — DMN Decision Model

## Por qué se hace

La implementación debe modelar decisiones.

DMN se puede usar para tres decisiones críticas:

1. ¿Qué herramienta usar para cada caso de uso?
2. ¿Puede un agente ejecutar una acción o requiere aprobación humana?
3. ¿Puede un caso de uso pasar a producción?

## Decisión DMN 1 — Herramienta por caso de uso

| Caso de uso | Datos requeridos | Acción autónoma | Sensibilidad | Herramienta recomendada |
|---|---|---|---|---|
| Reporting ejecutivo | M365 + documentos PMO | Baja | Media | M365 Copilot / ChatGPT Enterprise |
| Búsqueda de conocimiento | Documentación corporativa | Baja | Media | Glean / Gemini Enterprise / ChatGPT Company Knowledge |
| Delegar tareas multi-step | Email, calendario, docs | Alta | Media/Alta | Cowork / Claude Cowork con aprobación |
| Automatización con herramientas | Apps externas / APIs | Alta | Alta | MCP + gobierno + human approval |
| Desarrollo de soluciones IA | Código, specs, tests | Media | Media | Kiro / GitHub Copilot |

## Decisión DMN 2 — Acción autónoma vs aprobación humana

| Tipo de acción | Sensibilidad | Impacto | Confianza | Decisión |
|---|---|---|---|---|
| Crear borrador | Baja | Bajo | Alta | Permitir |
| Resumir información | Baja | Bajo | Alta | Permitir |
| Enviar correo | Media | Medio | Alta | Requiere aprobación |
| Modificar documento oficial | Alta | Alto | Media | Requiere aprobación PMO |
| Acceder a datos sensibles | Alta | Alto | Alta | Validar permisos |
| Ejecutar workflow externo | Alta | Alto | Media | Escalar a governance |
| Eliminar archivo | Alta | Alto | Alta | Bloquear o requerir aprobación explícita |

# 23. Paso 8 — AI Risk + Responsible AI

## Por qué se hace

Una Enterprise AI Platform puede aumentar productividad, pero también amplificar riesgos.

Riesgos típicos:

- Acceso indebido a documentos.
- Respuestas incorrectas con apariencia de autoridad.
- Prompt injection.
- Data leakage.
- Uso de herramientas no aprobadas.
- Shadow AI.
- Agent overreach.
- Acciones ejecutadas sin aprobación.
- MCP servers mal gobernados.
- Falta de auditoría.
- Vendor lock-in.
- Baja adopción.
- ROI no medido.

## Risk Register mínimo

| Riesgo | Causa | Impacto | Control | Evidencia |
|---|---|---|---|---|
| Data leakage | Permisos o conectores mal configurados | Alto | Data classification + least privilege | Access logs |
| Prompt injection | Documentos o fuentes maliciosas | Alto | Content filtering + source validation | Security review |
| Hallucination | Respuesta sin grounding | Medio/Alto | Citas obligatorias + revisión humana | Output review |
| Agent overreach | Agente ejecuta acciones fuera de alcance | Alto | Action boundaries + approval | Agent action log |
| Shadow AI | Usuarios usan herramientas no aprobadas | Medio/Alto | Catálogo aprobado + training | Adoption metrics |
| MCP risk | Servidores externos sin gobierno | Alto | MCP registry + security review | MCP inventory |
| Baja adopción | Falta de confianza o training | Medio | Champions + enablement | Usage analytics |
| ROI no capturado | Sin baseline ni owner | Medio/Alto | Value realization plan | Benefits dashboard |

## Responsible AI Controls

| Principio | Control | Evidencia |
|---|---|---|
| Transparencia | Respuestas con fuentes y explicación | Citation logs |
| Seguridad | Acceso por rol y revisión de conectores | Access review |
| Privacidad | Minimización y clasificación de datos | Privacy review |
| Accountability | Owner humano por decisión y agente | Decision log |
| Fairness | Revisión de criterios de priorización | Bias review |
| Reliability | Evaluación de calidad y monitoreo | Quality metrics |
| Human Oversight | Aprobación en acciones sensibles | Approval logs |
| Traceability | Registro de prompts, fuentes y acciones | Audit trail |

# 33. Nota de uso responsable

Antes de usar IA con datos reales de un cliente:

1. Verificar políticas internas.
2. Usar herramientas enterprise aprobadas.
3. Evitar datos personales o confidenciales si no existe autorización.
4. Validar permisos y clasificación de información.
5. Revisar conectores, apps y MCP servers.
6. Mantener supervisión humana en acciones sensibles.
7. Registrar decisiones críticas.
8. No aceptar recomendaciones de IA sin revisión profesional.
9. Consultar a Legal, Security, Privacy y Compliance cuando aplique.
10. Medir adopción y valor antes de escalar.
